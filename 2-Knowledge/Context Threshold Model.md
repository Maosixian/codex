# Context Threshold Model

## 定义

上下文不是越长越好。

当对话长度、工具输出、任务分支和历史噪音持续增长时，系统会先失去检索效率，再失去判断稳定性，最后出现 context rot。

因此 `codex` 需要把上下文管理当成一套显式模型，而不是临场感觉：
- 在还没腐烂前先整理
- 把该沉淀的内容移出当前对话，写回 vault
- 把当前对话压缩成足够继续工作的最小上下文

## 为什么要分层

如果只在“快满了”时一次性处理，通常已经太晚：
- 当前对话已经混入无关历史
- 关键结论被埋住
- 下一步判断成本明显上升

所以更稳的做法是分两层：

### 1. Soft threshold

接近 `70%` 时，默认进入轻压缩。

目标不是立刻换 session，而是：
- 提前降噪
- 提炼当前任务的工作记忆
- 把值得长期保存的内容写回对应层

### 2. Hard threshold

接近 `85%` 或已经出现明显 context rot 时，升级为强压缩。

目标是：
- 生成可续接 handoff
- 把当前对话缩到只保留继续工作所需的信息
- 默认建议切到新 session

## 判断信号

阈值判断采用双轨：

- Codex 负责近似判断：
  - 会话长度
  - 任务复杂度
  - 工具输出量
  - 分支数量
  - 是否开始反复重述或重新检索同一批内容
- Xavier 如果看到更准确的 `/usage` 或平台信号，以平台信号覆盖

## 信息去向

压缩不是只写一个摘要。

它至少涉及三件事：
- 当前对话内提炼最小工作集
- `4-Episodic/` 留下 restart trail
- 按 canonical home 更新 `5-Projects/`、`6-Users/`、`3-Practices/`、`2-Knowledge/`

## 成功标准

这套模型有效时：
- 新 session 不需要重放整段历史
- 长对话仍然能维持判断质量
- 重要结论不会只留在聊天里
- 上下文增长不会直接等于噪音增长
